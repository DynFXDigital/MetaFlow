# Phase 3: Launch Mechanics and Release Posture

## Goal

Finalize GitHub operational mechanics so an open-source launch is predictable, supportable, and low-noise.

## Scope

- Issue/PR intake quality.
- Release/tag discipline and notes quality.
- Marketplace + GitHub source-of-truth alignment.

## Work Items

- [ ] Verify issue templates and PR template still match updated README/support policies.
- [ ] Confirm release process documents `v0.x` versioning policy and preview semantics.
- [ ] Add/refresh launch checklist for:
  - Issues enabled
  - Discussions decision (enabled/deferred with rationale)
  - release tag flow (`v0.1.0`, `v0.2.0`, etc.)
  - release notes generation expectations
- [ ] Validate Marketplace publication prerequisites and token handling docs.
- [ ] Rehearse one preview release path (tag or manual dispatch) in a controlled run.

## Acceptance Criteria

- Templates, docs, and release workflow language are consistent.
- Maintainers can execute release path from docs without tribal knowledge.
- Public users can infer support and contribution pathways from repository landing pages.

## Evidence

- Updated release/runbook docs and template diffs.
- Link to successful rehearsal run (or dry-run evidence) in PR notes.

## Risks

- Rehearsal may be blocked by missing org-level secrets/permissions.
- Discussions moderation burden may exceed current maintainer capacity.
